# Как устроен SonLang изнутри (Под капотом)

Этот раздел предназначен для продвинутых разработчиков, которые хотят понять, как именно устроен и работает интерпретатор SonLang, написанный на языке программирования Rust.

---

## 1. Конвейер выполнения кода

Когда вы запускаете файл `.son`, интерпретатор выполняет три последовательных шага:

```
  Исходный код (.son) 
          │
          ▼
    [ Лексический анализатор (Lexer) ]  <── Разбирает текст на Токены
          │
          ▼
      [ Синтаксический анализатор (Parser) ] <── Строит дерево AST (Abstract Syntax Tree)
          │
          ▼
    [ Интерпретатор (Interpreter) ]  <── Выполняет AST-ноды шаг за шагом
```

### Шаг A: Лексический анализатор (`lexer.rs`)
Лексер читает ваш текстовый файл посимвольно и группирует буквы в логические "слова" — **Токены**.
Например, строка `пусть х = 5` превращается в список токенов:
1. `TokenKind::Let` ("пусть")
2. `TokenKind::Ident("х")` ("х")
3. `TokenKind::Assign` ("=")
4. `TokenKind::Number(5.0)` ("5")

### Шаг Б: Синтаксический анализатор (`parser.rs`)
Парсер берет плоский список токенов и превращает его в древовидную структуру — **Абстрактное синтаксическое дерево (AST)**.
Для строки `пусть х = 5` парсер создает AST-ноду:
```rust
Stmt::Let {
    name: "х".to_string(),
    value: Expr::Number(5.0)
}
```
AST-ноды в SonLang делятся на:
* **Инструкции (`Stmt`)** — действия, которые выполняются (объявление переменных, циклы, условия, вывод текста). Они не возвращают значений.
* **Выражения (`Expr`)** — вычисления, которые всегда возвращают какое-то значение (сложение, вызовы функций, обращение к переменным).

### Шаг В: Интерпретатор (`interpreter.rs`)
Интерпретатор рекурсивно обходит построенное дерево AST и выполняет соответствующие операции на Rust. 
* Переменные хранятся в специальной хэш-таблице `EnvMap`, обернутой в умные указатели `Rc<RefCell<EnvMap>>`, что позволяет передавать области видимости и организовывать замыкания для функций.

---

## 2. Управление памятью и Области видимости (Scope)

SonLang использует древовидную систему областей видимости.
Каждая функция или пользовательский модуль при запуске создает дочернее окружение (`EnvMap::child`), которое хранит ссылку на родительское окружение.

* **Чтение переменных**: Если переменная не найдена в текущей функции, интерпретатор автоматически спускается к родителю (в глобальную область) и ищет ее там.
* **Замыкания (Closures)**: Функции в SonLang запоминают окружение, в котором они были созданы. Это позволяет создавать функции внутри других функций, сохраняя доступ к их переменным.

---

## 3. Взаимодействие с Rust (Модульная система)

Все системные функции (графика, звук, системные вызовы) написаны на Rust в виде подключаемых плагинов.
Интерпретатор перехватывает вызовы вида `графика.окно(...)` на этапе выполнения выражения `Expr::MethodCall` и мгновенно перенаправляет аргументы в скомпилированный Rust-код. Благодаря этому графические окна работают с максимальной скоростью без накладных расходов виртуальной машины.
